메시징 디자인 패턴과 분산 시스템

February 15, 2018
programming design-pattern experience messaging
Read this in "about 2 minutes".

분산 시스템 소프트웨어를 개발할 때 겪게 되는 가장 큰 문제 중 하나는 분산된 프로세스들 사이에서 송/수신되는 다양한 메시지들을 효율적으로 다루는 것이다. 이 메시지들은 서로 다른 시점에 송/수신되며, 서로 다른 목적을 지니며, 서로 다른 포맷을 띤다. 이런 이질적인(heterogeneous) 메시지들을 어떻게 해야 효율적으로 다룰 수 있을까?

예를 들어, 분산 시스템 분야의 전통적인 문제인 분산 합의(Distributed consensus)를 살펴 보자. 분산 합의 문제의 목적은 고장(failure)를 일부 허용하는 시스템에서 프로세스들 사이의 일관된 상태를 유지하는 것이다. 문제 상황을 좀 더 구체적인 형태로 표현하면, 한 올바른 프로세스가 어떤 값을 수용하자는 제안(propose)을 다른 프로세스들에 알리게 되면, 나머지 프로세스들은 이 값을 결과적으로 수용하게 되어, 프로세스들 사이에서의 일관된 상태를 유지하게 된다. 간단해 보인다. 여기까지는 별 문제가 없어 보일 것이다. 하지만 앞서 언급한 상황에서 값을 제안하는 프로세스가 악의적인 해커에 의해 서로 다른 값을 서로 다른 프로세스에게 제안하자고 할 경우에는 “일관된 상태” 조건이 깨질 수 있다. 이는 소위 비잔틴 합의(Byzantine consensus) 문제라고도 불리는데, 이런 악의적인 프로세스가 일부 있음에도 올바른 프로세스들끼리는 일관된 상태를 유지하는 것이 비잔틴 합의의 목적이고, 이를 위해서는 특수한 프로토콜이 필요하다.

특수한 프로토콜이란, 크게 다음과 같은 절차로 구성된다.

  1. 시스템 내의 어떤 프로세스(A)가 값을 제안
  2. A가 제안한 값을 시스템 내의 나머지 프로세스들이 검증하는 절차
  3. 제안받은 값을 프로세스 상태에 반영할지 결정하는 절차.
    • 이 과정이 통과되면 각 프로세스는 해당 값을 프로세스 상태에 최종적으로 반영.

눈치가 빠른 독자는 위의 1,2,3이 PBFT의 Preprepare, Prepare, Commit 단계를 나타낸다는 것을 알아차렸을 것이다.

서론이 길어졌는데, 이제 다음과 같은 문제 상황을 생각해보자. 당신은 PBFT를 절실하게 개발하고픈 프로그래머다. PBFT를 개발하기 위해서는 이질적인 포맷을 가지는 메시지들을 일정한 순서대로 송/수신하고 각 메시지 유형에 따른 핸들러를 적당한 위치에 삽입해야 한다. 쉽게 생각할 수 있는 첫번째 솔루션은 분산 메시징 라이브러리를 사용해서 소켓 차원에서 헤더 구조체가 정의된 메시지를 송/수신하여, 헤더 내의 메시지 타입을 바탕으로 메시지를 핸들링해주는 방식일 것이다. 하지만 이 방식은 메시지 구조가 바뀜에 따라 추가적으로 작업해주어야 할 부분이 많고(Getter/Setter 작성 등), 유연성이 떨어지기 때문에 별로 추천되지 않는 방식이다.

두번째 방식은 메시지 포맷을 정의해주고 메시지 직렬화/비직렬화를 수행해주는 도구를 사용하는 것이다. 대표적으로 구글의 protobuf, flatbuffer 등이 있는데, 메시지 구조를 고수준에서 정의해주면 자동으로 코드를 원하는 프로그래밍 언어로 생성해준다. 이 방식을 택할 경우, 저수준 구조체로 정의된 메시지에 딸려오는 수동적이고 지루한 행위들을 피할 수 있고, 그만큼 에러가 발생할 확률도 훨씬 낮다.

하지만 protobuf 의 기능은 메시지 포맷을 개발자 관점에서 예쁘게 포장해주는 역할에 그친다. 이를 좀 더 잘 활용하기 위해서는 오래된 디자인 패턴 중 팩토리 패턴을 고려해볼 필요가 있다.

다음 포스팅에서는 팩토리 패턴을 활용한 메시지 팩토리 패턴에 대해 알아보고 이를 앞서 언급한 PBFT의 메시징 단계에 적용해보는 사례를 살펴볼 것이다.
